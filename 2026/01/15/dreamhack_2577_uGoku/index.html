II. The Text Segment Initial Analysis

There is only one process in .text segment, and seems no anti-debug. Therefore, we can try using Intel pin for some simple traces to roughly understand how the program runs. First, note that this program will print “wrong length“ if the length of input is wrong. We can easily test that the correct input length is 32. Then, the output will become “no“. Use the basic inscount0.so to count for the amount of instructions, and for wrong length, it shows 227917; however, for right length, this number booms to 22817615 — about 100 times higher. The whole instructions in text segment is about only 38000, so there must have someting to make this ‘function’ become a loop. Fortunately, this mechanism is quite obvious. Note that:

This part at the beginning of the entry point registered two handlers for two signals:

loc_8049137 for SIGILL (0x4) and loc_80490B0 for SIGSEGV (0xB). And at the end of the text segment:

There is an obvious illegal instruction at 0x807F23A. This will trigger the previous set handler, which can cause indirect jump (.plt:080490B6, loc_80490B0, SIGILL) or back to the beginning of the program (loc_8049137, SIGSEGV). Therefore, our next analysis direction seems to have become clearer: trace for the signals handler and indirect jump. Use the script:

Obviously, the ~1700 events in the center are highly repetitive, with only SIGILL running to the end of the code. Then purify and extract useful message from the whole trace:

From this we can notice that there is an encryption process of about 576 rounds in the whole program, and the SIGSEGV_JMP is actually used to call system functions. After 576 rounds of encryption, the next round may be used to compare the results, and after the failure, it puts ‘no’ and then exit.

III. The Text Segment Debug Analysis

Debug and set bp at the libc functions to record information:

At this moment, the .data:0x8081FB0 is the length of the input, and .data:0x8081FB8 & .data:0x8081FBC are pointed to input, input string is at .data:0x862AFBC. After that, the program loaded the string of wrong length and the addr of puts, but as the length is right, the SIGSEGV is not triggered. Then the memset was called:

This area happens to be the 0x100 bytes above of the input string. And in the following process, it is noticed that there are several frequently modified offsets that look like simulating registers in the data segment: .data:08081FB0, .data:08081FB4, .data:08081FB8, .data:08081FBC and .data:0862AE3C~0862AE58, .data:0862AE5C~0862AE78. The former appears to be registers within the loop (temporarily storing temporary variables used for calculations within the loop), while the latter are large registers across loops.

.data:0862AE7C looks like the loop count, as it incs every loop. After 4 loops, .data:0862AE3C and .data:0862AE5C becomes dword(input[3210]), and after another 4 loops, .data:0862AE40 and .data:0862AE60 becomes dword(input[7654]). Continue debugging:

Notice that this may be a doubule byte encryption, and the index order are shuffled. Write script to dump the 64-byte area every loop:

However, after analysis, in fact this encryption is a dual byte feistel + swap encryption: Every round choose 2 bytes, encrypt the first with the second, and the second gets the first’s value:

Not very surprisingly, the encrypt index list can be found in the program’s data segment: 0x8081A80 and 0x8081C80, both have 512 bytes, each round uses inp[0x8081A80[index]] and inp[0x8081C80[index]] to encrypt and swap them. Moreover, the purpose of other high entropy arrays surrounding these two arrays was initially unknown, but now they seem to exhibit characteristics strongly related to encryption and comparison processes:
